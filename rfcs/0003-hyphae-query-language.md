## Summary

Hyphae is the CSS-selector-inspired query language for Mycelium. This RFC
defines the grammar, implements a lexer (using the `logos` crate) and a
hand-written recursive-descent parser that emits an AST. Execution against
the Store is out of scope for this RFC and is deferred to RFC-0004 (Phase 2).

## Motivation

Mycelium's competitive moat is token efficiency: the Charter SLA requires
Hyphae queries to encode the same payload as JSON using ≤ 30% of the token
count. Until the DSL exists, the MCP server is nothing but a JSON-over-HTTP
wrapper with no competitive differentiator.

Phase 1 delivers the parse layer so:

1. The protocol shape is fixed and can be stabilised in MCP responses.
2. Tests validate the full surface area of the grammar before the executor
   (Phase 2) is written, following TDD discipline (Charter §5.1).
3. Editor tooling (syntax highlighting, completion) can be built on the AST
   independently from the execution engine.

## Detailed design

### Grammar (EBNF)

```ebnf
query      ::= selector_list EOF
selector_list ::= selector ( ',' selector )*
selector   ::= simple ( combinator simple )*
combinator ::= '>' | '~' | ' '   (* child, sibling, descendant *)
simple     ::= ( name_selector | kind_selector | universal ) pseudo_class*
name_selector ::= '#' IDENT
kind_selector ::= '.' IDENT
universal  ::= '*'
pseudo_class ::= ':' IDENT ( '(' selector_list ')' )?
IDENT      ::= [a-zA-Z_][a-zA-Z0-9_-]*
```

### Token types

| Token | Example | Description |
|---|---|---|
| `Hash` | `#login` | Name selector (`#` followed by identifier) |
| `Dot` | `.function` | Kind selector (`.` followed by identifier) |
| `Colon` | `:calls(` | Start of pseudo-class |
| `Gt` | `>` | Child combinator |
| `Tilde` | `~` | Sibling combinator |
| `Star` | `*` | Universal selector |
| `LParen` | `(` | Open pseudo-class argument |
| `RParen` | `)` | Close pseudo-class argument |
| `Comma` | `,` | Selector list separator |
| `Ident` | `function` | Bare identifier (used after `#`, `.`, `:`) |
| `Whitespace` | ` ` | Descendant combinator (significant between simples) |
| `EOF` | | End of input |

### AST types (Rust)

```rust
/// A complete Hyphae query: one or more comma-separated selector lists.
pub enum Ast {
    /// A comma-separated list of selectors: `#foo, .bar`
    SelectorList(Vec<Selector>),
}

/// A selector is a sequence of simple selectors connected by combinators.
pub enum Selector {
    /// A single simple selector: `#login`
    Simple(SimpleSelector),
    /// Two selectors joined by a combinator: `#foo > .bar`
    Combined {
        left: Box<Selector>,
        combinator: Combinator,
        right: Box<Selector>,
    },
}

/// Combinators between simple selectors.
pub enum Combinator {
    /// `>` — direct child
    Child,
    /// `~` — following sibling
    Sibling,
    /// ` ` (whitespace) — any descendant
    Descendant,
}

/// A simple (un-combined) selector, optionally decorated with pseudo-classes.
pub struct SimpleSelector {
    pub base: BaseSelector,
    pub pseudo_classes: Vec<PseudoClass>,
}

/// The base selector shape.
pub enum BaseSelector {
    /// `#name` — matches a specific symbol by name
    Name(String),
    /// `.kind` — matches all symbols of a given kind
    Kind(String),
    /// `*` — matches any symbol
    Universal,
}

/// A pseudo-class such as `:calls()` or `:imports(#foo)`.
pub struct PseudoClass {
    pub name: String,
    pub argument: Option<Box<Ast>>,
}
```

### Lexer

The lexer is generated by the `logos` crate. Whitespace is yielded as a
`Whitespace` token only when it appears between two non-whitespace tokens
(i.e. as a potential descendant combinator). Leading/trailing whitespace is
skipped. The logos skip attribute is used for that.

### Parser

A hand-written recursive-descent parser converts the token stream into the
AST. The parser is intentionally simple:

1. `parse_query` → parses a `SelectorList` (comma-separated).
2. `parse_selector` → parses a `Selector` (combinator-aware).
3. `parse_simple` → parses a `SimpleSelector` (base + pseudo-classes).
4. `parse_pseudo_class` → parses `:ident` or `:ident(selector_list)`.

### Error type

```rust
#[derive(thiserror::Error, Debug, PartialEq)]
pub enum ParseError {
    #[error("unexpected token `{0}` at position {1}")]
    UnexpectedToken(String, usize),
    #[error("unexpected end of input")]
    UnexpectedEof,
    #[error("lexer error at position {0}")]
    LexError(usize),
}
```

## Drawbacks

- The CSS-selector analogy is intentionally loose: Hyphae is not a DOM query
  language. Users familiar with CSS will find the surface familiar, but they
  must not assume full CSS semantics.
- `logos` adds a dependency. The crate is actively maintained, generates
  fast DFAs, and is already trusted in the Rust ecosystem (used by SWC,
  Oxc, etc.). It is lightweight (no runtime, codegen only).

## Alternatives

- **LALR parser via `lalrpop`**: More powerful for ambiguous grammars, but
  Hyphae's grammar is simple enough that a hand-written recursive-descent
  parser is more readable and easier to test. Rejected for added complexity.
- **Pest / `nom`**: Pest generates a parser from a grammar file (good for
  documentation), but the generated types are not as ergonomic. `nom`
  combinator chains are harder to read than recursive descent.  Both
  rejected in favour of the simpler, more explicit approach.
- **Full regex-based tokenizer**: Avoids a new dep, but `logos` generates
  faster code and the macro makes token definitions self-documenting.

## Prior art

- **CSS Selectors Level 4** (W3C): inspiration for `>`, `~`, ` `, `:pseudo()`
  syntax.
- **rust-analyzer's query API**: uses a similar selector-style API internally
  for IDE features.
- **tree-sitter queries** (`.scm`): pattern-based, but Lisp-syntax is
  unfriendly to AI agents in a prompt context.

## Migration

Phase 1 is purely additive (new crate, no public API removed). The placeholder
`Query` struct in `lib.rs` is superseded by the real types. No existing index
format is affected.

## Testing strategy

Per Charter §5.1, tests precede implementation.

- **Snapshot tests** (`insta`): ≥ 20 query strings covering all grammar
  productions. Each snapshot captures the debug representation of the AST.
- **Unit tests**: individual lexer token recognition; parser error paths.
- **Round-trip property test** (stretch goal): `proptest`-generated query
  strings are parsed and re-serialised; the re-serialised form re-parses
  to the same AST.

Acceptance criterion: all tests pass and `cargo clippy -- -D warnings` is
clean.

## Performance impact

Phase 1 is parse-only and introduces no hot-path changes to the Store.

| SLA | Current | After this RFC | Δ |
|---|---|---|---|
| Cold query | < 5 ms | < 5 ms | none |
| 3-hop traversal | < 1 ms | < 1 ms | none |
| Reactive refresh | < 10 ms | < 10 ms | none |
| Token efficiency | ≤ 30% JSON | ≤ 30% JSON (future) | none yet |

The DSL itself is the mechanism that will achieve the token efficiency SLA;
the executor (Phase 2) will measure this.

## Open questions

1. Should `#name` match case-insensitively? Proposal: case-sensitive,
   matching the underlying symbol names which are also case-sensitive.
2. Should the Hyphae grammar support attribute selectors (`[kind=function]`)
   in a future RFC? Yes — tracked as a future possibility.

## Future possibilities

- **RFC-0004 (Phase 2)**: `HyphaeEvaluator` — execute an `Ast` against the
  `Store` and return a `SymbolSet`.
- **Attribute selectors**: `[kind=function]`, `[file=~"src/auth"]`
- **Quantifiers**: `:nth-caller(2)`, `:min-depth(3)`
- **MCP integration**: MCP tools accept a Hyphae query string instead of
  (or in addition to) JSON parameters, achieving the token efficiency SLA.
